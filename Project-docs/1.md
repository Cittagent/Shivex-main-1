# FactoryOPS / Shivex Project Wiki

## 1. PROJECT OVERVIEW

### Project name and purpose

This repository is a Docker Compose driven industrial energy-intelligence platform branded as `Shivex` in the web and mobile clients. It ingests MQTT telemetry from edge devices, persists time-series telemetry to InfluxDB, projects low-latency device state into MySQL, evaluates alert rules, runs ML analytics, generates PDF reports, computes waste/loss views, exposes a natural-language copilot, and serves both a Next.js dashboard and an Expo mobile app.

The core business problem it solves is plant-floor observability for electrically monitored assets such as compressors:

- ingest telemetry in near real time
- expose fleet and per-device dashboards
- compute uptime, health score, energy, cost, and waste metrics
- raise rules/alerts from telemetry or time windows
- run anomaly/prediction/forecast jobs
- generate downloadable reports
- answer operator questions via an LLM-backed copilot

### Repo shape

This is a monorepo. Major applications/packages are:

- `/services/device-service` - asset registry, live projection, dashboard snapshots, health config, shifts, uptime, idle-running logic
- `/services/data-service` - MQTT ingestion, InfluxDB storage, telemetry query API, WebSocket streaming, DLQ, outbox relay, reconciliation, circuit breakers
- `/services/energy-service` - energy/loss projections, daily/monthly rollups, calendar APIs, live energy state
- `/services/rule-engine-service` - threshold/time-based rules, alert lifecycle, activity feed, email notification adapter
- `/services/analytics-service` - ML API and worker pair, dataset orchestration, training/inference jobs, artifact storage, retraining
- `/services/reporting-service` - consumption/comparison report generation, settings/tariff/notification APIs, scheduling, PDF creation
- `/services/waste-analysis-service` - waste analysis jobs, result storage, PDF/report object publishing
- `/services/data-export-service` - continuous and forced exports from InfluxDB to S3/MinIO datasets
- `/services/copilot-service` - AI copilot chat orchestration, read-only DB access, provider abstraction, service-context enrichment
- `/ui-web` - Next.js web dashboard
- `/shivex-mobile` - Expo React Native mobile app
- `/tools/device-simulator` - MQTT device telemetry simulator
- `/tests` - live-stack E2E verification and firmware validation assets

### Tech stack

- Backend language: Python 3.11
- Backend framework: FastAPI across all Python services
- Web frontend: Next.js 16, React 19, TypeScript 5, Tailwind CSS
- Mobile frontend: Expo 54, React Native 0.81, Expo Router
- Relational database: MySQL 8 (`ai_factoryops`)
- Time-series database: InfluxDB 2.7
- Object storage: MinIO / S3-compatible API
- Queue/cache/broadcast: Redis 7
- MQTT broker: EMQX 5
- ML/analytics libs: pandas, numpy, scikit-learn, XGBoost, TensorFlow, Prophet, SHAP
- PDF/reporting: WeasyPrint, Jinja2, matplotlib
- AI providers: Groq, Gemini, OpenAI
- Testing: pytest, pytest-asyncio, httpx, pytest-cov
- Deployment style: local/self-hosted Docker Compose stack

### Runtime / platform versions

- Python images: `python:3.11-slim`
- Node image: `node:20-alpine`
- MySQL image: `mysql:8.0`
- Redis image: `redis:7-alpine`
- InfluxDB image: `influxdb:2.7-alpine`
- MinIO image: `minio/minio`
- MQTT broker image: `emqx/emqx:5.3.0`

### Deployment target

The codebase is designed primarily for Docker Compose deployment. There is no GitHub Actions or hosted deployment configuration in the repo. Operationally, the intended boot path is `docker compose up -d --build`, with multiple services running Alembic migrations during container startup.

## 3. ARCHITECTURE OVERVIEW

### High-level design

The platform is a microservice-style Docker Compose system with a shared MySQL database and several specialist data stores:

- MySQL for relational application state and projections
- InfluxDB for immutable telemetry time series
- MinIO for datasets, reports, and dashboard snapshot payload offloading
- Redis for analytics queueing and real-time stream broadcast
- EMQX for MQTT telemetry ingress

### Core communication modes

- MQTT: device simulator/firmware -> `data-service`
- REST/HTTP: synchronous inter-service and UI/mobile access
- WebSocket/SSE-like streaming:
  - `data-service` exposes `/ws/telemetry/{device_id}`
  - `device-service` exposes `/api/v1/devices/dashboard/fleet-stream`
- Redis pub/sub/streams:
  - analytics queue
  - energy/device fleet live broadcaster channels
- S3/MinIO object storage:
  - analytics datasets and model artifacts
  - reporting PDFs
  - waste analysis PDFs
  - dashboard snapshots in device-service

### Service responsibilities

- `data-service`
  - receives telemetry from MQTT
  - validates/enriches payloads
  - writes time-series points to InfluxDB
  - enqueues downstream syncs in `telemetry_outbox`
  - relays outbox deliveries to device-service and energy-service
  - calls rule-engine for telemetry evaluation
  - exposes telemetry query APIs and WebSocket fan-out
  - runs DLQ retry and reconciliation jobs

- `device-service`
  - stores device metadata/configuration
  - maintains `device_live_state` projection rows
  - computes dashboard summary/fleet/live cards
  - manages shifts, health configs, idle config, device properties
  - materializes dashboard snapshots and monthly summaries
  - publishes fleet stream updates via Redis-backed broadcaster

- `energy-service`
  - consumes live-update payloads
  - computes daily/monthly energy and loss counters
  - maintains `energy_device_state` and fleet/day/month aggregate tables
  - exposes summary, today-loss, calendar, and per-device range APIs

- `rule-engine-service`
  - persists threshold and time-based rules
  - evaluates telemetry against rule definitions
  - creates alert rows
  - records activity events
  - optionally emits email notifications

- `analytics-service`
  - API role accepts jobs and exposes results/readiness
  - worker role executes ML jobs, writes results/artifacts, updates heartbeats
  - uses Redis queue backend by default and MinIO for datasets/artifacts

- `reporting-service`
  - builds energy and comparison reports
  - stores report metadata/results in MySQL
  - writes PDFs to MinIO
  - owns tariff and notification-channel settings APIs
  - runs report scheduler

- `waste-analysis-service`
  - orchestrates waste jobs
  - reads energy/device/reporting context
  - writes job rows and per-device summary rows
  - publishes downloadable files through MinIO presigned URLs

- `data-export-service`
  - continuously exports telemetry into object storage datasets
  - supports forced on-demand export windows
  - tracks checkpoints in MySQL

- `copilot-service`
  - wraps provider clients (Groq/Gemini/OpenAI)
  - builds domain-aware responses
  - now has a read-only MySQL engine path available in `src/database.py`

### Third-party / infrastructure integrations

- MySQL 8
- InfluxDB 2.x
- MinIO S3 API
- Redis 7
- EMQX MQTT broker
- SMTP for alert emails
- Groq / Gemini / OpenAI APIs for copilot

### Caching strategy

- Redis:
  - analytics job queue
  - energy/device live broadcaster channels
- in-process caches:
  - tariff caches
  - device meta cache
  - copilot/session-like UI memory only on clients
- frontend:
  - web uses direct fetch + local React state + adaptive polling; `no-store` for several dashboard calls
  - mobile uses React Query stale time + retries plus Zustand/secure-store for user identity choice

- MinIO buckets:
  - `energy-platform-datasets`

## 7. AUTHENTICATION & AUTHORIZATION

### Current state

There is no production-grade auth or authorization layer implemented across the stack.

Evidence:

- no JWT/session middleware in FastAPI services
- no auth provider integration in web/mobile apps
- route handlers generally accept anonymous requests
- several APIs use optional `tenant_id` as an input parameter rather than deriving tenant from identity

### What exists

- Mobile only:
  - `shivex-mobile/src/store/user.ts` stores `userName` and `userRole` in `expo-secure-store`
  - `shivex-mobile/app/_layout.tsx` redirects to `/role-select` if no local role is chosen
  - this is UX gating, not backend authorization

- Web only:
  - copilot page stores conversation history in `sessionStorage`
  - not an auth mechanism

### Effective permission model

- Everyone with network access can call most service APIs
- Tenant scoping is cooperative:
  - reporting and rule APIs accept `tenant_id`
  - device settings site waste config accepts `tenant_id`
- Copilot hardening added read-only DB connectivity, but not user-level access control

### Session/token strategy

- No httpOnly cookies
- No refresh tokens
- No JWT bearer token flow
- No OAuth/OIDC/Clerk/Auth.js

### Production implication

Auth is a major open gap and should be considered technical debt, not a completed feature.

## 9. KEY BUSINESS LOGIC

### Health score

Defined in `formulas.md` and device-service health config services:

- each configured telemetry parameter has normal and critical bounds plus weight
- score is weighted by per-parameter health
- zero values can optionally be ignored
- health score is only meaningful when configs exist for the device

### Uptime

Device uptime logic is shift-aware:

- planned runtime is derived from configured shifts
- maintenance breaks reduce planned/effective windows
- actual runtime comes from telemetry-derived running signal
- performance trend materialization buckets uptime into periodic rows

### Idle / unloaded / running detection

Device-service and energy-service use current/voltage/power heuristics:

- `running` when current/power signal is active beyond thresholds
- `idle` when inside shift and current is under idle threshold but still energized
- `unloaded` for special device state classification in live dashboard/device pages
- `off-hours` when energy is consumed outside shift windows

### Energy and loss calculation

Energy-service computes deltas from either monotonic counters or derived power:

- if `energy_kwh` style counter increases, use delta from counter
- otherwise derive `energy_delta = power_kw * dt_hours`
- split delta into:
  - idle
  - off-hours
  - overconsumption
  - total loss

### Optimistic locking

`device_live_state` and `energy_device_state` writes use optimistic locking helpers:

- read current `version`
- `UPDATE ... WHERE version=:expected_version`
- retry up to 3 times with 50 ms delay
- log warning on conflict, error on exhaustion
- skip rather than crash after retries fail

### Outbox pattern

`data-service` permanently fixed cross-store inconsistency:

- InfluxDB write first
- same MySQL transaction enqueues outbox rows
- background relay delivers to device-service/energy-service
- failures increment retries and dead-letter after max retries
- reconciliation checks Influx vs downstream freshness every 5 minutes

### Circuit breakers

Inter-service HTTP calls use a state machine:

- `CLOSED`: pass through
- `OPEN`: fail fast, return fallback/none, enqueue to outbox or DLQ as applicable
- `HALF_OPEN`: probe downstream after timeout

### Dashboard snapshots

Device-service periodically materializes snapshots:

- hot/fleet snapshots on a shorter interval
- energy/loss/monthly summary on a longer interval
- payload stored in MinIO with MySQL fallback
- optional one-time startup migration moves MySQL snapshot payloads into MinIO

### Reporting and waste jobs

- reporting marks stale jobs failed on startup if pending/processing older than 10 minutes
- waste-analysis does the same and also ensures a duplicate-job lookup index exists at startup

### Analytics role isolation

Analytics service enforces `APP_ROLE`:

- `api` role must not import ML-heavy libs or worker module
- `worker` role runs queue consumer and optional weekly retrainer

### Scheduled/background jobs

- device-service:
  - performance trend scheduler
  - idle aggregation
  - dashboard snapshot scheduler
  - live projection reconciler
  - optional snapshot migration on startup
- data-service:
  - MQTT consumption
  - DLQ retry worker
  - outbox relay worker
  - reconciliation worker
  - queue depth monitor and graceful drain
- reporting-service:
  - APScheduler report scheduler
- waste-analysis-service:
  - background job per waste request
- analytics-service:
  - Redis workers
  - weekly retrainer when enabled
- data-export-service:
  - continuous export worker

### Logging

- device-service uses Python logging plus `app/logging_config.py`
- data-service uses custom `src/utils/logging.py`
- analytics-service uses `structlog`
- reporting-service and waste-analysis-service use `logging.basicConfig`
- data-export-service sets up structured logging through `logging_config.py`

### What gets logged

- startup/shutdown lifecycle
- migration guard and startup retries
- circuit breaker transitions
- outbox delivery retries/dead lettering
- queue overflow and queue depth
- scheduler lag/warnings
- optimistic lock conflicts
- analytics stale job cleanup
- MinIO / DB / Influx readiness failures

### Errors currently swallowed or degraded

- web dashboard alert feed failures degrade to empty lists
- some startup checks log warning instead of crashing, e.g. waste Influx ping
- copilot returns user-friendly degraded responses on provider failure

## 14. LOCAL DEVELOPMENT SETUP

### Prerequisites

- Docker Desktop / Docker Engine with Compose
- Python 3.11 for local scripts/tests outside containers
- Node 20 if running `ui-web` outside Docker
- npm for web/mobile dependencies

### Zero-to-running flow

1. Ensure Docker is running.
2. Populate root `.env` if needed for AI keys and SMTP.
3. Start the entire stack:

```bash
docker compose up -d --build
```

4. Wait for infra and migrations.
5. Open:
   - web UI: `http://localhost:3000`
   - device-service docs: `http://localhost:8000/docs`
   - data-service health: `http://localhost:8081/health`

### Important startup behavior

Services that auto-run Alembic on startup:

- device-service
- analytics-service
- rule-engine-service
- reporting-service
- waste-analysis-service
- energy-service also runs `alembic upgrade head` in `start.sh`

### Local simulator

- Dockerized stack already includes EMQX
- simulator tooling:

```bash
docker compose exec -d tools-simulator python main.py
```

or use scripts such as `scripts/simulatorctl.sh`

### Common gotchas

- `docker compose up -d --build` does not reset volumes
- `docker compose down -v` resets named volumes, but services/simulators can repopulate fresh data immediately
- dashboard/UI can show stale client-side state until a hard refresh or refetch
- several services depend on others becoming healthy; startup order matters
- sample secrets in compose are development-only

## 16. KNOWN ISSUES & TECH DEBT

### TODO / FIXME scan

No meaningful `TODO`/`FIXME` markers were found in the source tree outside incidental mentions of log levels or test config.

### Technical debt identified from code

- No real authentication/authorization layer across services or clients.
- Multi-tenancy is soft and input-driven rather than identity-enforced.
- The repo still contains a legacy multi-database bootstrap script in `/db/bootstrap.sql`, while the live stack uses a unified `ai_factoryops` schema.
- Reporting and waste services use broad permissive CORS.
- Some services duplicate health endpoint semantics at both top-level and prefixed routers.
- The web app relies heavily on local component state and direct `fetch`, with no centralized cache invalidation or schema client.
- The workspace currently contains checked-in generated artifacts (`ui-web/.next`, node_modules trees in clients, pytest caches).
- Compose contains development credentials in plain text.
- Copilot has provider abstraction and DB hardening pieces, but no end-user auth or audit trail.

### Commented-out code blocks

No large dead/commented-out implementation blocks stood out in the audited source files; most code paths are live.

### Performance bottlenecks / operational risks

- Shared MySQL schema across many microservices increases blast radius for schema or lock issues.
- No API gateway/rate limiting layer found.
- Rule, reporting, waste, and analytics all stack on the same MySQL and object storage substrate.
- Frontend can momentarily present stale UI state between refetches/stream events.

## 17. GLOSSARY

- `device_live_state` - device-service live projection row used for fleet/dashboard views
- `energy_device_state` - energy-service live accumulation row for energy/loss counters
- `DLQ` - dead-letter queue for failed telemetry syncs and some recovery workflows
- `Outbox` - MySQL durability pattern used in data-service for downstream sync consistency
- `fleet snapshot` - paginated dashboard card representation of all devices
- `fleet stream` - event stream of fleet updates emitted by device-service
- `idle running` - energy consumed while the device is energized below productive threshold
- `off-hours` - energy consumed outside configured shift windows
- `overconsumption` - energy attributed to current draw above a configured threshold
- `health config` - per-parameter ranges and weights used to compute device health score
- `performance trends` - periodic materialized records for uptime/health trend charts
- `analytics job` - anomaly/prediction/forecast execution tracked in `analytics_jobs`
- `reporting schedule` - recurring report definition in `scheduled_reports`
- `snapshot` - cached dashboard payload, now typically stored in MinIO with pointer in MySQL
- `tenant_id` - soft multi-tenant scoping identifier used in reporting/rules/settings APIs
- `APP_ROLE` - analytics-service role selector: `api` or `worker`
- `schema_version` - telemetry schema version tag stored in InfluxDB
- `enrichment_status` - telemetry enrichment status tag stored in InfluxDB
